Dapr Document Extraction

upload → pub/sub → durable workflow → structured output  ·  ~/scratch/dapr-workflow

Two small Python services turn any document into schema-shaped JSON. The upload service accepts a file plus a JSON schema, parks the file in blob storage, and fires a pub/sub message. The process service picks that message up and runs a Dapr Workflow of durable, retryable activities: fetch the document, convert it to markdown locally, split it into chunks when it exceeds the LLM token budget, extract fields with OpenAI structured outputs (one parallel call per chunk, merged afterwards), and write everything back to storage. If anything crashes mid-job, the workflow resumes exactly where it left off.

The big picture

topology

Every box below is a separate process. The two FastAPI apps never talk to Redis, Azurite, or each other directly — all traffic goes through their Dapr sidecars, which resolve the blobstore binding, the pubsub component, and the workflow engine from YAML config. Swap Azurite for a real storage account or Redis for Service Bus and the Python code doesn't change.

Dapr building blocks

components/*.yaml

Dapr is configured entirely through three component files in components/, loaded by both sidecars via the Multi-App Run template. The application code only ever references component names (blobstore, pubsub) — never connection strings.

bindings.azure.blobstorage

blobstore

Output binding for the documents container. The endpoint field points it at the local Azurite emulator; in Azure you'd drop that line and change the credentials. Operations used: create, get, list.

pubsub.redis

pubsub

Message broker between the two services, using the Redis container that dapr init installed. Delivery is at-least-once — the design accounts for duplicates (§8).

state.redis

statestore

Backs the workflow engine. Dapr Workflow is built on actors, so this component must declare actorStateStore: "true" — it's where every workflow's history and progress live, and it's what makes crash recovery possible.

spec:

  - name: accountName

  - name: accountKey

  - name: containerName
    value: documents            # created automatically if missing
  - name: endpoint
    value: http://127.0.0.1:10000   # ← the Azurite switch
  - name: decodeBase64
    value: "true"               # every 'create' sends base64 → stored as raw bytes
components/statestore.yaml — the workflow-critical line
spec:
  type: state.redis
  version: v1
  metadata:
  - name: redisHost
    value: localhost:6379
  - name: actorStateStore
    value: "true"              # without this, workflows won't start
Why decodeBase64? Binding payloads travel as JSON, and raw PDF/DOCX bytes aren't valid JSON strings. So every upload base64-encodes the content, and the component decodes it before writing the blob — the stored file is byte-identical to the original.
03

Step 1 — Upload takes the job in

upload/app.py · :8100
upload → pub/sub → workflow → activities

POST /upload is a multipart form: file (the document) and schema (the JSON schema that will drive extraction later). The handler does exactly three things — validate, store, publish — and returns immediately. It never touches OpenAI or MarkItDown; the heavy lifting is someone else's job.

upload/app.py — POST /upload (trimmed)FastAPI + DaprClient
@app.post("/upload")
async def upload(file: UploadFile = File(...), schema: str = Form(...)):
    schema_obj = json.loads(schema)          # 400 if not valid JSON
    content = await file.read()

    job_id = uuid.uuid4().hex                # one id for everything
    ext = pathlib.Path(file.filename or "document").suffix or ".bin"
    doc_blob = f"{job_id}/original{ext}"

    with DaprClient() as client:
        # 1 · store the document via the blobstore binding
        client.invoke_binding(
            binding_name="blobstore", operation="create",
            data=base64.b64encode(content),          # decodeBase64 undoes this
            binding_metadata={"blobName": doc_blob},
        )
        # 2 · tell the process service there is work
        client.publish_event(
            pubsub_name="pubsub", topic_name="process-topic",
            data=json.dumps({
                "job_id": job_id, "doc_blob": doc_blob,
                "file_name": file.filename, "schema": schema_obj,
            }),
            data_content_type="application/json",
        )
    return {"job_id": job_id, "status_url": f"http://localhost:8101/status/{job_id}"}
design choice

The message is self-contained

The document travels by reference (blob name), but the schema travels inline in the message. That makes the pub/sub payload the complete workflow input — the process service needs nothing else to run the whole job, even days later.

the payload

What lands on the topic

{
  "job_id":   "8c15e2c8d5ce4243…",
  "doc_blob": "8c15e2c8…/original.html",
  "file_name": "invoice.html",
  "schema":   { "type": "object", … }

Step 2 — Pub/sub hands the job over

process/app.py · subscriber
upload → pub/sub → workflow → activities

Dapr wraps every published message in a CloudEvent envelope and delivers it as an HTTP POST to whoever subscribed. The dapr-ext-fastapi extension registers the subscription declaratively — the sidecar discovers it by calling GET /dapr/subscribe at startup.

process/app.py — the subscriberidempotent by design
dapr_app = DaprApp(app)   # registers GET /dapr/subscribe for the sidecar

@dapr_app.subscribe(pubsub="pubsub", topic="process-topic")
def on_process_request(event_data=Body()):
    payload = event_data.get("data")        # unwrap the CloudEvent envelope
    if not payload or "job_id" not in payload:
        logger.error("dropping malformed message")
        return {"success": True}            # ack → no poison-message loop

    try:
        DaprWorkflowClient().schedule_new_workflow(
            workflow=doc_processing_wf,
            input=payload,
            instance_id=payload["job_id"],   # dedup key!
        )
    except Exception as e:
        if "already exists" in str(e).lower():
            # pub/sub is at-least-once; a redelivery is fine — just ack it
            logger.info("duplicate delivery, workflow already scheduled")
        else:
            raise                           # 500 → Dapr redelivers later
    return {"success": True}
The idempotency trick. Redis pub/sub through Dapr guarantees at-least-once delivery — the same message can arrive twice. Because the workflow instance id is the job id, a second delivery can't start a second job: the engine rejects the duplicate instance and the handler acks it. Exactly-once effects from at-least-once delivery, in four lines.

Step 3 — The durable workflow

process/workflow.py
upload → pub/sub → workflow → activities

The orchestrator is a Python generator: each yield ctx.call_activity(...) hands control to the engine, which runs the activity, records the result in Redis, and resumes the generator with the return value. The workflow function itself must stay deterministic — all real work (I/O, LLM calls, file conversion) lives in the activities.

process/workflow.py — the orchestratordeterministic, replayable
wfr = WorkflowRuntime()

@wfr.workflow(name="doc_processing_wf")
def doc_processing_wf(ctx: DaprWorkflowContext, wf_input: dict):
    job_id = wf_input["job_id"]
    # activities exchange BLOB NAMES, never content -- state stays tiny
    meta = yield ctx.call_activity(retrieve_document,
        input={"doc_blob": wf_input["doc_blob"]}, retry_policy=retry_policy)

    markdown_blob = yield ctx.call_activity(convert_to_markdown,
        input={"job_id": job_id, "doc_blob": wf_input["doc_blob"], ...})

    chunk_blobs = yield ctx.call_activity(plan_chunks,
        input={"job_id": job_id, "markdown_blob": markdown_blob})

    if len(chunk_blobs) == 1:   # fits the token budget: single call
        extraction = yield ctx.call_activity(extract_structured,
            input={"markdown_blob": chunk_blobs[0], "schema": wf_input["schema"]})
    else:                       # map-reduce: parallel fan-out, then merge
        tasks = [ctx.call_activity(extract_structured,
                     input={"markdown_blob": c, "schema": wf_input["schema"]})
                 for c in chunk_blobs]
        partials = yield when_all(tasks)
        extraction = yield ctx.call_activity(merge_extractions,
            input={"partials": partials, "schema": wf_input["schema"]})

    blobs = yield ctx.call_activity(write_results,
        input={"job_id": job_id, "extraction": extraction})

    return {"job_id": job_id, "chunks": len(chunk_blobs), **blobs}
how durability works

Event sourcing + replay

Every activity result is appended to the instance's history in the Redis actor state store. If the process dies after step 2, the restarted engine re-executes the generator from the top — but yields for completed activities return instantly from history instead of re-running. Execution fast-forwards to the first unfinished step and continues. That's why the orchestrator must be deterministic: replay has to take the same path every time.

per-step retries

The shared RetryPolicy

RetryPolicy(
  first_retry_interval=timedelta(seconds=2),
  max_number_of_attempts=3,
  backoff_coefficient=2,      # 2s → 4s → 8s
  max_retry_interval=timedelta(seconds=30),
  retry_timeout=timedelta(minutes=5),
)

Applied to every call_activity. A flaky OpenAI call or a brief Azurite hiccup is absorbed silently; only after 3 failed attempts does the workflow itself fail — with the error captured in failure_details.

Step 4 — The activities, map-reduce style

process/workflow.py · the actual work
upload → pub/sub → workflow → activities

Activities are plain Python functions — free to do I/O, call APIs, block. They exchange blob references, never document content: each activity reads what it needs from storage and writes its own output back. Only small values (blob names, metadata, the extraction JSON) enter workflow state, so history stays tiny no matter how big the document is.

activity 1 · blob binding

retrieve_document

data = _get_blob(act_input["doc_blob"])   # binding 'get'
return {"size": len(data)}                # metadata only — never the bytes

Verifies the original upload is actually retrievable from Azurite and fails fast if not. Only the document's metadata enters workflow state.

activity 2 · local conversion

convert_to_markdown

content = _get_blob(act_input["doc_blob"])   # reads storage itself
result = MarkItDown().convert(tmp_path)
markdown_blob = f"{job_id}/converted.md"
_create_blob(markdown_blob, result.text_content.encode(), "text/markdown")
return markdown_blob                          # a name, not 850 KB of text

MarkItDown (Microsoft) converts pdf / docx / pptx / xlsx / html to markdown — fully local, no API. The markdown goes straight back to blob storage; only its name is returned.

activity 3 · the splitter

plan_chunks

budget_chars = int(EXTRACT_MAX_TOKENS * CHARS_PER_TOKEN)
if len(markdown) <= budget_chars:
    return [act_input["markdown_blob"]]   # no split needed
# else: cut at paragraph boundaries, write {job}/chunks/chunk-NNN.md
return chunk_blobs

Guards the LLM's input limit (the full Odyssey is 305k tokens — over gpt-5-mini's 272k). Oversized markdown is split into token-budgeted chunk blobs; small documents pass through untouched.

activity 4 · the LLM step (× N, parallel)

extract_structured

markdown = _get_blob(act_input["markdown_blob"]).decode()
response_format = {
  "type": "json_schema",
  "json_schema": {
    "name": "extraction",
    "schema": act_input["schema"],  # the user's uploaded schema
    "strict": True,
  },
}
resp = client.chat.completions.create(
    model=os.getenv("OPENAI_MODEL", "gpt-5-mini"),
    messages=[...], response_format=response_format)
return json.loads(resp.choices[0].message.content)

OpenAI structured outputs: the model is constrained to emit JSON matching the schema. Strict mode requires additionalProperties: false and all properties in required; if the user's schema doesn't comply, the activity catches the 400 and retries with strict: False. The key comes from process/.env via python-dotenv. Documents beyond the token budget (200k, configurable) are split by a plan_chunks activity; the workflow fans out one extraction per chunk in parallel (when_all) and a merge_extractions activity unifies the partial results.

activity 5 · the reducer

merge_extractions

partials = act_input["partials"]   # one JSON per chunk, via when_all
return _call_openai(
    "Merge these partial extractions from consecutive segments "
    "of ONE document into a single result conforming to the schema...",
    user=json.dumps(partials), schema=act_input["schema"])

Runs only when there was more than one chunk. The partial JSONs are small, so this call is cheap; it unions list fields and keeps document order.

activity 6 · persist results

write_results

result_blob = f"{job_id}/extracted.json"
_create_blob(result_blob,
    json.dumps(act_input["extraction"], indent=2).encode(),
    "application/json")
return {"result_blob": result_blob}

Lands the extraction next to the original and the markdown (already written in step 2) — the job folder is now complete and every artifact is inspectable.

Why references and not content? A war story. An earlier version passed base64 document bytes and markdown text through workflow state. It worked for small invoices — then a 1.7 MB PDF (the full Odyssey) was killed mid-workflow, and on resume the replay history had grown to 4.46 MB: past the 4 MB gRPC work-item limit between sidecar and worker. The engine could never deliver the orchestration again and the instance wedged permanently. With blob references, history stays a few KB no matter the document size.

Reliability — what happens when things break

the reason for the workflow

The whole point of running the pipeline as a Dapr Workflow instead of a plain message handler: every failure mode below has a defined, tested behavior instead of a lost job.

Failure scenarioOutcomeWhat happens
OpenAI timeout / 429 / transient error Self-heals The retry policy re-runs just that activity: 2s → 4s → 8s backoff, 3 attempts. The other steps never re-execute.
Process service crashes mid-job Resumes History lives in Redis, not in the process. On restart the engine replays the orchestrator, fast-forwards through completed steps, and continues from the first unfinished activity.
Same pub/sub message delivered twice Deduplicated instance_id = job_id — the second schedule_new_workflow raises "already exists", the handler acks, no duplicate job runs.
Subscriber raises an unexpected exception Redelivered The HTTP 500 tells the sidecar delivery failed; Dapr redelivers the message later. Combined with the dedup above, this is safe.
Malformed message on the topic Dropped + logged Acked deliberately so a poison message can't loop forever; the error is in the process service logs.
All 3 attempts of an activity fail FAILED, visibly The workflow ends in FAILED; GET /status/{job_id} returns the exception message in failure. The original document is still in blob storage — fix the cause and re-submit.
Try it. Upload a big document, kill the process app mid-run (dapr stop -f dapr.yaml), start it again — the job completes without re-uploading anything. The status endpoint shows the same instance picking up where it stopped.
